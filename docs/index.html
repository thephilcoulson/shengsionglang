The thesis

Operating a supermarket network is a compilation problem. Inventory, ordering, delivery routing and replenishment are rules that should be written once, checked ahead of time, and executed at native speed — not scripted, not interpreted, not glued together in spreadsheets.

What we built

  • A domain-specific language (.sheng) whose primitives are supermarket operations
  • A compiler (shengc) that lowers programs to native machine code
  • Errors caught at compile time; execution is a plain native binary

Why it matters to the board

  • Store logic is portable & auditable source, not tribal knowledge
  • Zero runtime footprint — deploys anywhere libc runs
  • Correctness guarantees before a single unit ships

Compilation pipeline

program.sheng ──▶ shengc ──▶ C source ──▶ cc/clang ──▶ ./program (native ELF)
             │                                             │
  lexer │ parser │ codegen                      runs on the CPU
Front end
Hand-written scanner → recursive-descent parser → typed AST. Deterministic, single pass, precise line diagnostics.
Back end
AST → portable C11. The system C compiler does register allocation, inlining, and machine codegen — we inherit decades of optimization.
Output
A self-contained ELF binary. ldd → libc only. No Python, no JVM, no bytecode loop.

Compiler is itself ~1,310 lines of C in src/shengc.c — the classic bootstrap path (cf. early Go, Nim, V).

Domain nouns (store/warehouse/supplier/truck/product) and verbs (order/deliver/stock/sell/price/restock/report) are language keywords — not library calls.

Value & type model


typedef enum { V_NIL, V_INT, V_FLOAT,
               V_BOOL, V_STR } VType;
typedef struct {
  VType t;
  long long i;   /* exact unit counts   */
  double    f;   /* prices, ratios      */
  int       b;
  char     *s;   /* entity names        */
} Value;
  • Quantities are integers (long long) — no fractional units of stock; enforced at the boundary.
  • Money/ratios are doubles with money-safe formatting.
  • Mixed arithmetic promotes int→float only when needed; 10/2 → 5, 7/2 → 3.5.
  • Division/modulo by zero, overselling, over-capacity delivery → trapped.

Functions compile to C functions (native recursion & calls). Control flow compiles to native if/while. Entity ops compile to direct calls into the embedded engine — no dispatch loop, no interpretation overhead.

The supply-chain engine

Emitted verbatim into every binary — a complete inventory system in portable C.

Entities → structs in dynamic arrays
Loc, Prod, Supp, Truck, Rule. O(1) amortized inserts; linear scans that the C compiler vectorizes.
Operations → pure functions
do_stock, do_sell, do_order, do_deliver, add_rule — each with invariant checks (no negative inventory, capacity limits).
Auto-restock is event-driven — check_rules() runs on every stock mutation. Falling below a threshold triggers an immediate reorder. No polling, no scheduler, deterministic.

static void check_rules(const char*prod,const char*store){
  for(int i=0;i<g_nrules;i++)
    if(!strcmp(g_rules[i].prod,prod) && !strcmp(g_rules[i].store,store)){
      Loc*L=find_loc(store);
      if(loc_qty(L,prod) < g_rules[i].threshold)
        loc_add(L,prod,g_rules[i].qty);   /* replenish */
    }
}

Correctness: two lines of defense

Compile time — rejected before deploy

  • Undefined variables
  • Unknown entity properties (milk.color)
  • Non-literal entity config
  • Invalid assignment targets
  • Lex & parse errors, with line numbers
exit ≠ 0, no binary produced

Run time — trapped, never silent

  • Overselling stock you don't have
  • Delivery exceeding truck capacity
  • Drawing more than a source holds
  • Division / modulo by zero
  • Dynamically-built unknown product/location
aborts with a diagnostic
A tree-walking reference interpreter (Python, 100% statement + branch coverage) pins the intended semantics; the native compiler is validated against it.
